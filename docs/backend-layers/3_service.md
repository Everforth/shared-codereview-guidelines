## Service層（ビジネスロジック、データ処理 等）

### 必須チェック項目：

- **ビジネスロジックの正確性**: データの加工、計算、判定処理が仕様通りに実装されているか
- **トランザクション境界**: データベーストランザクションが適切なスコープで設定されているか
- **データ変換処理**: EntityからDTOへの変換、外部APIレスポンスの変換が適切か
- **Service間の依存関係**: 他のServiceへの依存が適切で、循環参照を避けているか
- **不要なモジュールのインポート削除**: 使用していないモジュールやライブラリがインポートされていないか
- **エラーハンドリング**: ビジネスロジック内で発生するエラーを適切に処理し、例外をスローしているか（HTTPステータスはControllerで決定）
- **責務分離と可読性**: Serviceがビジネスロジックのみに集中し、HTTP関連の処理を含んでいないか。また、似た名前の変数・メソッドがある場合は使い分けが命名から明確に判別できるかを同時に確認する
- **重複コードの排除**: 類似のロジックが既存のコードに存在しないか確認し、再利用可能な共通関数やServiceを活用しているか
- **非同期処理**: Promiseを返す処理に`await`が適切に付与されているか
- **暗黙的な前提条件の排除**: メソッドが外部での事前チェックを前提としていないか、単体で正しく動作するか
- **nullチェックの適切な配置**: null checkを追加する前に、呼び出し元でnullを排除できないか確認する
- **any型の禁止**: 型を明示的に定義し、any型の使用を避ける
- **不適切なasキャストの回避**: 暗黙的な前提に基づいて型推論エラーを握りつぶさず、正規の方法で値を取得する
- **DTOにロジックを含めない**: データ変換やフィルタリングはServiceの責務であり、DTOのコンストラクタで行わない
- **カラム名とデータの整合性**: カラム名が示す意味と実際に保存されるデータを一致させる
- **命名による責務の明確性**: 今回追加された名前が既存の類似した名前と責務の違いを判別できるかを確認する。判別できない場合は🔴修正必須として指摘する
- **設定値の明示化**: `undefined` や `null` が「デフォルト動作」「処理スキップ」など特別な意味を持たないこと。意図を表す明示的な値（専用の型、Enum、定数）を使用し、設定漏れを型で検出できるようにする
- **関心の分離**: 外部システムとの契約（メタデータキー、イベント型など）とビジネス設定（どのエンティティをどう処理するか）は別ファイルに分離する。1つのファイルに複数の関心が混在している場合は修正を求める

### パフォーマンス考慮点：

- バッチ処理での一括操作
- 適切なキャッシュ戦略（TTL設定）
- 不要なDB呼び出しの削減
- upsertパターンの適切な使用: race condition対策の細かいコードは全体に広げず、本当にhigh volumeが来る箇所のみに限定する

### TypeORM使用方針：

- QueryBuilderよりRepository APIを優先: QueryBuilderは型推論が効かないため、複雑なクエリが必要な場合を除きRepository APIを使用する
- `repository.save()` や `entityManager.save()` が含まれる場合、「🟨 確認」として指摘する。内部的にN+1クエリを発行するため、ボリュームが多い場合はQueryBuilderの使用を検討すべきであることを伝える。
- ページネーションライブラリの活用: パフォーマンス最適化が必須の箇所を除き、`nestjs-typeorm-paginate`等の標準ライブラリで保守性を確保する

### 設計・命名規則：

- 特殊な実装には必ずコメントで理由を明記
- モジュール間で名前の衝突を避ける
- 型変換ロジックの配置: 型の変換を担うロジックは、その型を定義したファイル内で責任を持つ（transform関数として型定義ファイルにexport）
- 類似メソッドの使い分け明示: 似たような名前のメソッドがある場合、どのような状況で使い分けるかが命名から具体的にわかるようにする（例: `getUser` vs `getUserWithPermissions`、`save` vs `saveWithValidation`）

### この層で扱わないこと：

- Entity定義、データベーススキーマ（DB層の責務）
- TypeORMのクエリビルダーの書き方の詳細（DB層の責務、Serviceでは「どのデータを取得するか」のみ）
- DTOの定義とバリデーション（Controller層の責務）
- HTTPリクエスト/レスポンス処理（Controller層の責務）
- HTTPステータスコードの決定（Controller層の責務）
- 認証・認可ガードの設定（Controller層の責務）
- ルーティング、エンドポイント定義（Controller層の責務）
